## 2. 项目当前定位

这个项目是一个“心理/情绪记录”方向的微信小程序，当前产品主链路已经跑通：

1. 用户在输入页输入一段情绪文字
2. 小程序调用云函数做真实 AI 情绪分析
3. 返回结构化情绪结果
4. 结果页展示为“情绪日记”
5. 分析成功后自动保存为云端日记
6. 用户可以在详情页、日历页、统计页查看自己的历史记录
7. 用户系统已经接入微信身份（基于 `openid`），并支持账号资料页

项目当前不再是纯原型，而是已经进入“可云端运行、可多设备读取自己数据”的阶段。

### 7.3 已创建的云函数

当前已存在 4 个业务云函数：

- [cloudfunctions/emotionAnalyze](D:/毕设项目/毕设小程序/cloudfunctions/emotionAnalyze)
- [cloudfunctions/journalService](D:/毕设项目/毕设小程序/cloudfunctions/journalService)
- [cloudfunctions/userBootstrap](D:/毕设项目/毕设小程序/cloudfunctions/userBootstrap)
- [cloudfunctions/userProfile](D:/毕设项目/毕设小程序/cloudfunctions/userProfile)

### 7.4 各云函数职责

#### emotionAnalyze

职责：

1. 接收前端原始文本
2. 调用真实 AI（DeepSeek）
3. 做结果结构整理
4. 做高风险兜底
5. 返回统一分析结果

#### journalService

职责：

1. 创建云端日记
2. 查询最近记录
3. 按月列出记录
4. 获取详情
5. 生成月度统计
6. 删除日记（软删除）
7. 迁移旧本地记录

采用 `action + payload` 路由方式。

#### userBootstrap

职责：

1. 通过 `OPENID` 建立或获取用户档案
2. 返回当前用户资料
3. 在用户首次进入时完成建档

#### userProfile

职责：

1. 更新当前用户昵称
2. 更新当前用户头像 `fileID`
3. 返回更新后的用户资料

### 7.5 云函数部署注意

每次新增或改动云函数后，需要在微信开发者工具里对对应目录执行：

`创建并部署：云端安装依赖（不上传 node_modules）`

尤其这几个函数：

- `emotionAnalyze`
- `journalService`
- `userBootstrap`
- `userProfile`

最近一次联调里，`userProfile` 出现过 `FUNCTION_NOT_FOUND`，说明：

1. 前端代码已调用它
2. 但云端未识别到对应函数
3. 交接后第一件事建议重新部署一次 `userProfile`

## 8. 用户系统与云端日记存储

### 8.1 设计原则

当前用户系统不是传统注册登录，而是轻量微信身份：

1. 不做手机号注册
2. 不做用户名密码
3. 只基于 `openid` 识别用户

### 8.2 数据库集合

云数据库当前应有两个集合：

1. `users`
2. `journals`

### 8.3 权限策略

两个集合的权限应该设置成：

`所有用户不可读写`

原因：

1. 不让前端直接读写数据库
2. 所有业务统一走云函数
3. 由云函数按 `openid` 控制数据归属

### 8.4 users 字段语义

当前 `users` 主要字段包括：

- `openid`
- `nickname`
- `avatarUrl`
- `createdAt`
- `lastLoginAt`
- `localMigrationDone`
- `journalCount`

说明：

1. `avatarUrl` 现在实际存的是云存储文件 `fileID`
2. `nickname` 是用户资料显示名称
3. 数据归属仍然只靠 `openid`

### 8.5 journals 字段语义

当前 `journals` 主要字段包括：

- `openid`
- `clientMigrationKey`
- `rawInput`
- `mainEmotion`
- `subEmotions`
- `explanations`
- `foreignEmotionWord`
- `analysis`
- `suggestion`
- `isNegative`
- `isHighRisk`
- `diaryText`
- `source`
- `createdAt`
- `dateKey`
- `updatedAt`
- `isDeleted`
- `deletedAt`

说明：

1. `displayDateTime` 已被移除，不再作为正式存储字段
2. `diaryText` 保留，用于未来导出日记
3. `source` 用来区分 `ai` / `mock`
4. `isDeleted + deletedAt` 支撑软删除

### 8.6 已配置索引

控制台中已经配置过这组索引：

#### users

- `openid`

#### journals

- `openid + clientMigrationKey`
- `openid + isDeleted + createdAt`
- `openid + isDeleted + dateKey`

这套索引已经足够支撑：

1. 用户查找
2. 首页最近记录
3. 日历按月查询
4. 本地旧记录迁移去重

## 16. 我做过的重要稳定性修复

这个项目早期出现过多次：

1. `app.json` BOM 导致 JSON 解析失败

3. 某些 JS 文件因编码残留导致模块注册失败

处理方式：

1. 多个文件统一转为 UTF-8 无 BOM
2. 对关键启动链路文件重写，减少中文编码污染

### 16.2 模块注册失败修复

出现过：

- `module 'config/env.js' is not defined`
- `module 'utils/time.js' is not defined`
- `module 'services/userService.js' is not defined`

处理方式：

1. 调整配置文件位置
2. 重写关键模块
3. 对启动链路使用更稳妥的 ASCII / Unicode 转义形式

### 16.3 详情页误显示失败

详情页以前请求还没回来时，会先显示“没有找到这条日记”。

后来改为显式三态：

1. `loading`
2. `succeeded`
3. `failed`

### 16.4 账号资料保存链路问题

已发现并处理/部分处理的问题：

1. `showLoading` / `hideLoading` 配对警告  
   已通过移除多余全局 loading 收口。

   已通过前端并发锁和点击态保护处理。
5. 云端历史头像 URL 过期后 403  
   已通过 `avatarDisplayUrl` 解析与失败回退处理。
